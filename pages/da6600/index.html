工厂模式

# 应用场景，什么时候该用工厂模式？

如果创建对象逻辑不复杂，直接用new即可。创建逻辑复杂的情况：

  • 类似规则匹配，代码中存在if-else，动态的根据不同的类型创建不同的对象。
  • 单个对象本身创建复杂，各种组合，初始化。

# 工厂模式的优点：

  • 封装变化
  • 代码复用
  • 隔离复杂性
  • 控制复杂度

# 工厂模式分类：

  • 简单工厂（Simple Factory）：也叫静态工厂方法。
    • 把对象的创建逻辑独立成一个工厂类。
    • 如果对象可复用，可用map将其缓存起来。
  • 工厂方法（Factory Method）：将if逻辑去除，利用多态。但使用时可能又有许多if-else，因此可以为工厂类再创建一个简单工厂，工厂的工厂，用来创建工厂类对象。
  • 抽象工厂（Abstract Factory）：复杂，不常用。让一个工厂负责创建多个不同的类型对象。

复杂度无法被消除，只能被转移：

  • 不用工厂模式，if-else 逻辑、创建逻辑和业务代码耦合在一起
  • 简单工厂是将不同创建逻辑放到一个工厂类中，if-else 逻辑在这个工厂类中
  • 工厂方法是将不同创建逻辑放到不同工厂类中，先用一个工厂类的工厂来得到某个工厂，再用这个工厂来创建，if-else 逻辑在工厂类的工厂中

# 代码

//简单工厂模式，方法1
public class RuleConfigParserFactory {
  public static IRuleConfigParser createParser(String configFormat) {
    IRuleConfigParser parser = null;
    if ("json".equalsIgnoreCase(configFormat)) {
      parser = new JsonRuleConfigParser();
    } else if ("xml".equalsIgnoreCase(configFormat)) {
      parser = new XmlRuleConfigParser();
    }...
    return parser;
  }
}

////简单工厂模式，方法2（对象可复用时，去除了if-else）
public class RuleConfigParserFactory {
  private static final Map<String, RuleConfigParser> cachedParsers = new HashMap<>();
  static {
    cachedParsers.put("json", new JsonRuleConfigParser());
    cachedParsers.put("xml", new XmlRuleConfigParser());
    ...
  }
  public static IRuleConfigParser createParser(String configFormat) {
    IRuleConfigParser parser = cachedParsers.get(configFormat);
    return parser;
  }
}

//工厂方法，新增一种parser时，只需新增一个实现的factory就行了。
public interface IRuleConfigParserFactory {
  IRuleConfigParser createParser();
}

public class JsonRuleConfigParserFactory implements IRuleConfigParserFactory {
  @Override
  public IRuleConfigParser createParser() {
    return new JsonRuleConfigParser();
  }
}
...
//使用
public RuleConfig load(String fileExtension) {
    IRuleConfigParserFactory parserFactory = null;
    if ("json".equals(fileExtension)) {
        parserFactory = new JsonRuleConfigParserFactory();
    }...
    IRuleConfigParser parser = parserFactory.createParser();
    return parser.parse();
}

# 工厂模式与依赖注入容器有何区别？

DI容器底层的设计思路就是基于工厂模式。DI容器相当于一个大的工厂类，负责在程序启动时，根据配置事先创建好对象。当程序需要使用某个类的对象时，直接从容器中获取。因为持有一堆对象，才称为容器。

工厂类只负责某个类或一组相关类对象的创建。DI容器负责整个应用中所有类对象的创建，且负责的内容也比较多。

# spring 使用到的工厂模式

BeanFactory 采用的是简单工厂模式。

FactoryBean 采用的是工厂方法模式。

# DI容器的核心功能

有3个核心功能：

  • 配置解析。BeanConfigParser，负责将配置文件解析为BeanDefinition结构。
  • 对象创建。通过反射机制，将所有类对象的创建都放在一个工厂类中完成，如BeansFactory，根据BeanDefinition创建对象。
  • 对象生命周期管理

# BeansFactory中的createBean()是一个递归函数，如何解决循环依赖问题

Spring解决循环依赖的办法是多级缓存。

构造器初始化方式，无法解决循环依赖

set注入方式初始化，有两种:

  • 第一种，创建的是单例对象，可以解决。
  • 第二种，创建的是原型对象，由于DI容器不缓存对象导致无法提前暴露一个创建中的对象，故无法解决。
